## 原理：

此时登有tk账号的浏览器会启动（自动填写提示词和自动上传图片）这个过程是在另一台电脑上完成的，目前为了测试那台电脑默认只开了2个浏览器。它们会同时消费这个任务列表里的任务。当视频生成好后，不出意外的话会自动的把视频下载到NAS本地网络共享盘里。 视频下载位置： TkGravity\AI素材\tk-auto下载好的视频会以 task_${taskId}.mp4 命名

每一条视频对应一个任务id，可以用任务列表里的任务id对应 TkGravity\AI素材\tk-auto\taskId.mp4

消费者会间隔2分钟轮询任务列表任务，执行步奏「清空→填图→填超长提示词→提交」
暂时无法在飞书文档外展示此内容
因 ProseMirror 会拦截回填文字操作 fill 通常不生效，所以采用 模拟逐字输入, 延迟10微秒。

## 注意：

不排除少数视频会生成失败，目前视频生成失败后可手动点击重试，后续会根据记录排查失败原因。
失败原因：

1. 登录会话过期

浏览器需要记住登录会话，这个会话是在一定时间有效期。需要手动登录，涉及：（动态验证码）于此同时后台任务再跑它不知道TK账号会话是否有效，因为没打通TK API，此时没登陆态找不到输入框和上传按钮，程序会报错，这个报错不影响主程序，只是在会话过期时先前的任务会失败，需要在会话可用时手动重新生成
暂时无法在飞书文档外展示此内容
2. TK登录需要代理，自动化电脑需要实时跑代理，确保程序能打开并登录到对应的页面执行自动化操作
3. DOM定位出错，由于依赖浏览器元素和标签定位，有可能TK在发新版本时该元素和标签有变化，甚至布局有变化此时对应的自动化工具要做相应调整
4. 模型拉胯切换账户调整默认模型

针对上述问题：1、2 需要手动维护（到下载机子手动在会话过期手动登录，确保代理正常运行，页面正常访问） 这个只要机子不关只需要做一次操作，具体还得看TK登录会话过期时间,  3 定位出错就要持续优化工具代码

待开发功能评估：
目前只支持浏览器打开态，任务消费者定时2分钟检测待销费任务列表，消费完任务浏览器空余时间不关闭，因为分页加载只有8-10条视频，浏览器消费任务可能非常多，关闭浏览器等待下载者再次打开后要加载多少条视频列表未知，涉及到滚动条上滑加载，目前最好策略停留在当前浏览器上，用已有视频列表状态
